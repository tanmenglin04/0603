# 代码审查角色定位

## 概述

本项目采用双角色代码审查机制：高级前端工程师和高级后端工程师在每次 bug 修复或 feature 迭代后进行协作审查，确保代码改动不引入新的问题。

---

## 快速使用

```
审查触发 → 前后端分工审查 → 汇总问题 → 评分过滤 → 问题修复 → 复查确认
```

**审查命令**：直接粘贴代码改动或描述改动内容，AI 将按照以下清单进行审查。

---

## 审查触发条件

### 必须审查
- [ ] Bug 修复完成后
- [ ] Feature 迭代完成后
- [ ] 代码重构完成后
- [ ] 合并到主分支前

### 建议审查
- [ ] 新增接口（前端调用路径变化）
- [ ] 新增依赖包
- [ ] 配置文件变更
- [ ] 数据库 schema 变更

## 角色一：高级前端工程师

### 核心技术栈

- Vue 3.4.15（Composition API + `<script setup>`）
- Vite 5.0.12 构建工具
- Vue Router 4.2.5 路由管理
- Axios 1.6.5 HTTP 客户端
- ECharts 5.4.3 数据可视化
- CSS 变量与 scoped 样式

### 审查清单（置信度评分）

每项问题需标注置信度（0-100），≥80 分才纳入报告：

| 置信度 | 含义 | 示例 |
|-------|------|------|
| 0-25 | 误报，不值得提出 | linter 能捕获的问题、代码风格偏好 |
| 50 | 中等确信，可能是真问题 | 边界条件未充分验证 |
| 75 | 高度确信，是真问题 | 内存泄漏风险、N+1 查询 |
| 100 | 绝对确信，必须修复 | 安全漏洞、数据不一致 |

#### 1. 功能正确性（置信度基准：75+）
- [ ] **问题**：页面数据渲染是否符合后端接口契约
- [ ] **问题**：列表页/详情页/表单页的边界条件是否处理（空数据、加载中、错误状态）
- [ ] **问题**：路由跳转逻辑是否正确，参数传递是否完整
- [ ] **问题**：页面刷新后状态是否能正确恢复（如 localStorage 中的用户信息）
- [ ] **问题**：退出登录后是否正确清理状态并跳转

#### 2. 接口契约（置信度基准：80+）
- [ ] **问题**：API 调用路径是否与后端接口一致
- [ ] **问题**：请求方法（GET/POST/PUT/DELETE）是否正确
- [ ] **问题**：请求参数和请求体格式是否匹配后端 DTO
- [ ] **问题**：响应数据解包是否正确（是否正确处理 `Result<T>` 包装）
- [ ] **问题**：错误响应是否正确解析并展示友好提示

#### 3. 状态管理（置信度基准：80+）
- [ ] **问题**：`ref` / `reactive` 状态初始化是否合理
- [ ] **问题**：异步请求后状态更新是否正确触发视图更新
- [ ] **问题**：是否有内存泄漏风险（定时器、事件监听器未清理）
- [ ] **问题**：`localStorage` 读写是否正确处理序列化/反序列化
- [ ] **问题**：Token 过期或无效时是否正确引导用户重新登录

#### 4. 性能（置信度基准：75+）
- [ ] **问题**：是否存在不必要的重复渲染
- [ ] **问题**：大列表是否使用虚拟滚动或分页
- [ ] **问题**：ECharts 图表实例是否正确销毁和重建
- [ ] **问题**：生产构建是否正确 tree-shaking

#### 5. 安全（置信度基准：90+）
- [ ] **问题**：敏感信息是否硬编码在代码中
- [ ] **问题**：XSS 风险：用户输入内容是否正确转义
- [ ] **问题**：是否存在调试代码（如 `console.log`）遗留在生产代码
- [ ] **问题**：API 请求是否正确携带 Authorization header

#### 6. 代码风格（置信度基准：50+）
- [ ] **问题**：是否使用 Composition API 而非 Options API
- [ ] **问题**：组件是否使用 `scoped` 样式避免污染
- [ ] **问题**：是否存在重复代码可以提取为公共组件
- [ ] **问题**：注释是否完整，关键业务逻辑是否有说明

### 误报过滤（不纳入报告的问题）

以下问题**不应**在审查报告中提出：
- linter、typechecker、compiler 能自动捕获的问题
- 纯代码风格偏好（无性能/安全影响）
- 预存问题（非本次改动引入）
- 已被 lint ignore 注释抑制的问题
- 整体功能的合理变更（非本次改动目标）

## 角色二：高级后端工程师

### 核心技术栈

- Spring Boot 2.7.18
- Spring Data JPA + Hibernate 5.6
- MySQL 8 数据库
- JWT (io.jsonwebtoken:jjwt 0.11.5) 认证
- BCrypt 密码加密
- Maven 构建

### 审查清单（置信度评分）

#### 1. 架构设计（置信度基准：75+）
- [ ] **问题**：Controller 层的职责是否单一（是否承担了 Service 的职责）
- [ ] **问题**：Service 层是否正确处理业务逻辑和事务边界
- [ ] **问题**：Repository 层是否正确使用 JPA 方法命名约定
- [ ] **问题**：是否有不必要的循环依赖
- [ ] **问题**：配置是否外部化（无硬编码值）

#### 2. 业务逻辑（置信度基准：80+）
- [ ] **问题**：参数校验是否完整（`@Valid` / `@Validated` 使用是否正确）
- [ ] **问题**：异常处理是否完整（`GlobalExceptionHandler` 是否覆盖所有异常类型）
- [ ] **问题**：事务边界是否正确（增删改操作是否有 `@Transactional`）
- [ ] **问题**：并发场景是否处理（如库存扣减、状态流转）
- [ ] **问题**：边界条件是否处理（空集合、零值、最大值等）

#### 3. 数据完整性（置信度基准：90+）
- [ ] **问题**：实体字段是否与数据库表结构一致
- [ ] **问题**：外键关联是否正确（`TeamMember.projectId` → `Project.id`）
- [ ] **问题**：级联操作是否正确定义（删除项目时是否处理团队成员）
- [ ] **问题**：数据初始化逻辑（`DataInitializer`）是否确定性
- [ ] **问题**：是否存在数据竞争或不一致风险

#### 4. 安全（置信度基准：90+）
- [ ] **问题**：密码是否使用 BCrypt 加密存储
- [ ] **问题**：JWT Secret 是否足够复杂且不硬编码
- [ ] **问题**：敏感接口是否有权限控制（仅管理员可操作的接口是否校验 role）
- [ ] **问题**：SQL 注入风险（JPA `findBy` 方法是否安全）
- [ ] **问题**：异常信息是否泄露内部实现细节

#### 5. API 设计（置信度基准：75+）
- [ ] **问题**：RESTful 接口命名是否规范（资源复数名词）
- [ ] **问题**：HTTP 方法使用是否正确（GET 查、POST 增、PUT 改、DELETE 删）
- [ ] **问题**：响应格式是否统一（`Result<T>` 包装）
- [ ] **问题**：错误码是否准确（`ResultCode` 枚举是否覆盖所有场景）
- [ ] **问题**：分页接口是否支持（如数据量大时的处理）

#### 6. 性能（置信度基准：80+）
- [ ] **问题**：是否存在 N+1 查询问题（`findAll()` 是否需要 JOIN FETCH）
- [ ] **问题**：数据库索引是否合理（`ProjectRepository.findByStatus()` 是否用到索引）
- [ ] **问题**：HikariCP 连接池配置是否合理
- [ ] **问题**：是否有不必要的数据库查询（如循环内查询）
- [ ] **问题**：日志是否过多影响性能

#### 7. 代码质量（置信度基准：50+）
- [ ] **问题**：是否有未处理的异常（空 catch 块或 catch 后无操作）
- [ ] **问题**：是否正确使用 `@Slf4j` 日志记录关键操作
- [ ] **问题**：是否有重复代码可以提取为公共方法
- [ ] **问题**：方法长度是否合理（过长方法应拆分）
- [ ] **问题**：注释是否完整（类注释、方法注释、关键逻辑说明）

#### 8. 测试覆盖（置信度基准：50+）
- [ ] **问题**：Service 层是否有单元测试
- [ ] **问题**：异常场景是否有测试覆盖
- [ ] **问题**：数据初始化逻辑是否有测试

### 误报过滤（不纳入报告的问题）

以下问题**不应**在审查报告中提出：
- linter、compiler 能自动捕获的问题
- 预存问题（非本次改动引入）
- 通用代码质量建议（除非明确写在 CLAUDE.md 中）
- 已被 `@SuppressWarnings` 抑制的问题
- 整体功能的合理变更（非本次改动目标）

## 协作审查流程

### 标准流程（5 步）

```
1. 触发审查
   ↓ 开发者提供代码改动描述或文件路径
2. 前后端分工
   ↓ 前端工程师审查前端改动
   ↓ 后端工程师审查后端改动
3. 置信度评分
   ↓ 每个问题标注 0-100 置信度
4. 过滤与汇总
   ↓ 过滤 <80 分的问题
   ↓ 汇总 ≥80 分的问题为审查报告
5. 问题修复与复查
   ↓ 开发者修复问题
   ↓ AI 复查确认
```

| 特性 | CODE_REVIEW_ROLES.md | 官方 `/code-review` 插件 |
|-----|---------------------|------------------------|
| **适用场景** | 本地代码、非 Git 仓库 | GitHub PR 自动化审查 |

## 问题描述写作指南（测试思维）

### 核心原则：说"人话"

代码审查是给开发人员看的，但问题描述要让**不懂代码的人也能看懂**。想象你是在向产品经理描述一个 bug。

### 好问题三要素

```
1. 场景（用户视角）
   "用户在登录页面输入正确的用户名密码后"
   
2. 操作（具体步骤）
   "点击登录按钮，或者按回车"
   
3. 结果（实际现象 + 期望现象）
   "页面没有跳转，一直在转圈" vs "应该跳转到首页"
```

### 模板

```markdown
**场景**：...
**操作**：...
**实际结果**：...
**期望结果**：...
```

### 示例对比

| ❌ 差描述（开发视角） | ✅ 好描述（测试视角） |
|---------------------|---------------------|
| "findById 抛出 USER_NOT_FOUND 异常" | "查询项目详情时，项目不存在却报'用户不存在'，误导排查" |
| "未使用 @Transactional" | "用户注册时，数据库保存了用户名但没保存密码，因为事务没生效" |
| "N+1 查询问题" | "打开项目列表页要 5 秒，因为每个项目都单独查了一次负责人" |
| "Cors 配置不安全" | "黑客可以模拟任意来源的请求，存在被 CSRF 攻击风险" |
| "异常处理器被注释" | "后端报错时前端只显示'网络错误'，看不到具体原因" |

### 禁止用语

- ❌ "违反了 DRY 原则" → 改为 "这段代码和另一处一模一样，改一个要改两个"
- ❌ "存在内存泄漏风险" → 改为 "页面来回切换几次后，浏览器越来越卡"
- ❌ "未做空值判断" → 改为 "后端返回空数据时，前端页面直接白屏"

### 置信度与描述详细程度对照

| 置信度 | 描述要求 |
|-------|---------|
| 90-100 | 必须包含**完整的触发步骤** + **截图/日志**（如有） |
| 75-89 | 必须包含**触发步骤** + **期望/实际结果** |
| 50-74 | 描述**现象**即可，根因分析可选 |
