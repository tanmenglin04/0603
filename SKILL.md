# 扫描项目生成提示词 Skill

## 核心原则：先扫现有项目，再出题

本 Skill 的 **首要工作是扫描用户当前指定的仓库/工作区里的真实代码与配置**；十二道提示词必须是「扫出来的」，不是凭常识或模板杜撰。若用户未指定路径，**默认以当前工作区/仓库根为扫描范围**。

## 何时启用

- 用户要求「扫描现有项目」「扫描项目」「根据代码/仓库出题」「生成一套提示词/题目」。
- 需要固定数量：5 道 Bug 修复、5 道 Feature 迭代、1 道代码理解、1 道代码重构、1 道代码工程化、1 道代码测试。

## 目标

在 **已对现有项目完成一轮以上实际阅读**（读文件、看目录、必要时跑只读检索）的前提下，产出 **恰好 12 条** 可执行提示词；每条须 **锚定仓库内真实文件/模块/行为**。
禁止在未扫描的情况下用占位路径凑满十二道题；信息不足时宁可写清「待查阅 `xxx` 后细化」，也不要虚构不存在的路径。

## 执行流程

1. **确认扫描对象（现有项目）**
   - 明确根目录：用户给出的路径，或当前打开的仓库根。
   - 在「扫描摘要」里写清：本轮扫了哪些顶层目录与关键文件。
   - 确认 Repo ID（自选 Repo 或题库 Repo）
2. **建立扫描范围**
   - 读 `README`、包管理文件、主要源码目录与关键配置（如路由、入口、`package.json`、`pyproject.toml` 等）。
   - 若有 monorepo，说明聚焦哪个子包，或说明跨包关系。
3. **抽取事实**（仅来自当前仓库，用于出题）
   - 模块边界、数据流、外部依赖、错误处理缺口、测试覆盖强弱处。
   - 记录 **仓库内真实路径**（相对根目录）与 **函数/类名**（若适用）。
4. **按配额生成**（见下一节）
   - 缺信息时在输出里列出 **假设** 或 **待确认项**，仍尽量基于已读代码命题；不得编造不存在的文件路径。
5. **自检**
   - 数量：5+5+1+1+1+1=12。
   - 每条提示词包含：背景、具体目标、约束、可验收标准（或检查清单）。
   - **可溯源**：每道题至少一处可对应到已扫描范围内的具体位置或行为。

## 各类题目的内容规范

### Bug 修复（×5）

每条应基于 **可观察迹象**：潜在空指针、竞态、错误被吞、边界条件、类型不安全、资源泄漏、与文档不一致的行为等。
提示词须写明：**复现或推断路径**、**涉及文件/符号**、**期望修复后发生什么**、**不要破坏的契约**。

### Feature 迭代（×5）

五条难度或影响面宜有区分（例如：小改进 / 中等功能 / 与集成或 UX 相关的扩展）。
写明：**用户场景**、**输入输出/API**、**与现有模块的衔接**、**回滚或特性开关**（若适用）。

### 代码理解（×1）

要求是 **解释型任务**：例如梳理一次请求的调用链、某个状态机的转移条件、配置如何覆盖默认值。
须指定 **起点**（入口文件或公开 API）和 **要回答的问题列表**。

### 代码重构（×1）

目标为 **行为不变** 的可维护性提升：拆分模块、消除重复、澄清命名、收窄类型、提取策略对象等。
写明：**当前痛点**、**禁止改变的外在行为**、**建议的验收方式**（如关键用例或对比清单）。

### 代码工程化（×1）

目标为 **工程化改进**：CI/CD 流程、构建优化、依赖管理、代码规范、自动化工具接入等。
写明：**当前工程痛点**、**改进收益**、**建议的工具或流程**。

### 代码测试（×1）

与仓库测试风格一致（单元 / 集成 / 合约 / E2E 择一或组合说明）。
写明：**要覆盖的代码路径**、**fixture 或 mock 策略**、**断言要点**、**如何本地运行**。

## 提示词编写规范

1. **首轮提示词**：可逻辑详尽、篇幅较长，提供足够上下文
2. **后续轮次提示词**：需客观、简明地描述实际未完成的功能或 Bug
3. **可验证性**：问题必须在项目运行环境中可验证
4. **严禁**：脱离实际的细节问题、机器味过重的 AI 生成话术
5. **难度要求（核心约束）**：所有提示词必须满足「5 轮对话门槛」，即首轮无法完成，需经过 5 轮以上多步骤思考、验证、修复、回归才能交付
6. **难度梯度**（Bug修复和Feature迭代全部使用复杂提示词，确保5轮对话才能完成）：
   - **Bug 修复（5道全部为复杂题）**：
     - 每道题均涉及多文件关联、隐式依赖或边界条件
     - 需要跨模块排查（数据库 + 业务逻辑 + 异常处理）
     - 需要分析调用链、线程安全、事务边界或外部服务依赖
     - 首轮仅能定位问题区域，详细修复需要多轮推理和验证
     - **禁止**：给出具体修复方案、明确指出修改哪一行代码
     - **必须包含**：至少 2 个需要独立验证的子问题、1 个潜在的回归风险、1 个边界条件测试场景；任务描述中不得出现具体文件路径、函数名、变量名等代码标识符，统一使用业务语义或模块级别的自然语言描述
   - **Feature 迭代（5道全部为复杂题）**：
     - 每道题均涉及新增 API、修改已有逻辑、多表关联或前后端协同
     - 需要设计数据模型、处理事务边界、保证向后兼容
     - 需要补充单元测试、集成测试或 E2E 测试
     - 首轮仅能理解需求框架，详细实现和联调需要多轮迭代
     - **禁止**：给出具体实现方案、API 设计草稿或代码框架
     - **必须包含**：至少 1 个需要与现有系统联调的验证点、1 个向后兼容方案、1 个性能/并发测试场景；任务描述中不得出现具体文件路径、函数名、变量名等代码标识符，统一使用业务语义或模块级别的自然语言描述
   - **代码理解题**：需要阅读多个模块并梳理完整调用链路，**禁止直接给出完整调用链图**，需分模块描述并逐步关联
   - **代码重构题**：涉及跨模块依赖提取、消除重复代码、测试覆盖补全，**禁止给出重构后代码**，仅描述目标状态和约束
   - **工程化题**：需要完善 CI/CD 流程、多环境配置、依赖安全审计，**禁止给出完整配置文件**，仅描述痛点和改进方向
   - **测试题**：需要 Mock 外部依赖、处理异步场景、补充边界用例，**禁止给出测试代码**，仅描述覆盖目标和 fixture 策略
7. **满意率控制（强制）**：题目难度必须确保 AI 在第 5 轮对话才能完成任务，"首轮即完成"或"4 轮内完成"的比例严格控制在 **0%**，即所有题目必须天然需要 5 轮以上才能完成

## 写作约束

- **命题来源**：题目必须能从「已扫描的现有项目」中对应到具体代码或配置；通用面试题、与当前仓库无关的臆造场景一律不要。
- 使用中文撰写题目正文（路径与标识符保持原样）。
- 每条提示词尽量 **自包含**，使他人无需再翻聊天记录即可执行。
- 避免重复的五道题指向同一小段代码；若不可避免，注明 **递进关系**。
- 不要泄露真实密钥；示例中用占位符。

## 最小示例（单条格式示意）

以下为 **一条复杂 Bug 题** 的粒度示例（真实执行时应替换为项目真实路径与行为），每道题都必须按此复杂度设计：

> **背景**：用户缓存模块在缓存未命中时仍返回上一用户的显示名。日志显示问题发生在获取当前登录用户信息服务时。进一步排查发现该服务同时被鉴权模块和管理后台模块调用，且存在多实例部署时的缓存竞态问题。
> **任务**：修复缓存键与请求上下文不一致的问题；补充与身份令牌会话绑定相关说明；处理多实例部署时的缓存同步问题；需要评估对现有功能的影响并提供兼容方案。
> **约束**：不改变对外接口响应结构；如需迁移，提供兼容层；当前测试覆盖率仅 40%，需补充测试；需要处理分布式缓存与本地缓存的失效策略不一致问题。
> **分步验收（5轮门槛）**：
>
> - 第 1 轮：定位缓存键生成逻辑，确认与身份令牌 header 的关联方式，分析多实例部署场景下的缓存一致性问题，识别所有调用链路上可能受影响的服务
> - 第 2 轮：设计分布式缓存失效方案，评估集群缓存 vs 单机缓存的取舍，产出 2 套以上候选方案并列出各自风险点
> - 第 3 轮：实现缓存键生成修复，实现分布式缓存失效机制，编写数据迁移脚本，验证修复后同一用户不同令牌不再混淆
> - 第 4 轮：补充单元测试和集成测试，覆盖缓存未命中、并发写入、缓存穿透等场景；进行全链路压测，验证向后兼容
> - 第 5 轮：全量回归测试通过，文档更新，编写运维手册，处理边界告警和监控告警配置
